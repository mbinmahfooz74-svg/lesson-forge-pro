2 · Your weekly rhythm

The platform is designed so your steady-state effort is read one email, clear one queue.

Monday (~20 minutes)

  1. Read the advisory email (also shown on your dashboard home). It tells you what the scout found, what was drafted, what's waiting for you, and the estimated AI spend.
  2. Open the Review queue. Approve the weekly briefings you're happy with (approval publishes them to subscribers), reject noise, and approve any new-topic proposals worth teaching — approving a topic automatically plans its course.
  3. Spot-check one generated pack if content changed significantly.

After teaching a session (~5 minutes)

Go to Feedback, pick the vertical, paste your session transcript and/or debrief notes. The Feedback Analyzer distills them into skill memory — next week's drafts inherit what it learned. Every learning step is versioned and visible on the same page.

Anytime

  • Drop URLs into the Source pipeline when you find good material — everything downstream gets smarter.
  • Press Run weekly cycle on the dashboard if you don't want to wait for Monday 06:00 UTC.

4 · The content lifecycle

  1. Vertical — create it with a name + one sentence. The Architect builds taxonomy, glossary, search queries, and files 6–8 course proposals into your Review queue.
  2. Sources — drop URLs. The quality gate scores each capture; weak ones are flagged, never silently indexed.
  3. Course — plan from any topic (or approve an Architect/Scout proposal). You get a multi-session arc with Bloom's-mapped objectives.
  4. Draft — per session, the Drafter writes the minute-by-minute educator guide, grounded in your ingested sources and your skill memory.
  5. Materials — one click renders the pack: slides (PPTX), assessment + rubric (DOCX), learner handout (DOCX + PDF). Tenant packs carry the tenant's branding. Arabic editions come from the Localizer.
  6. Publish — sessions with materials are PUBLISHED and appear in the subscriber Library; approved briefings appear in the Briefings feed and go out in digests.
  7. Learn — your feedback updates skill memory; the next cycle drafts closer to your voice.

5 · Review queue & the autonomy dial

Every vertical has an autonomy setting:

ModeBehavior
Review-first (default)Briefings and proposals wait in your Review queue. Nothing reaches subscribers without your click.
Full-autoThe cycle publishes briefings directly; you see what happened in the advisory. Graduate a vertical here once you trust its output.
Locked rule — InvestmentThe Investment vertical is compliance-locked: review-first, and every briefing/course carries the "educational, not financial advice" disclaimer in both languages. The Briefing Writer is constrained against personalized buy/sell recommendations. Don't relax this without a compliance review.
Current limitationThere's no UI toggle for the autonomy dial yet — all verticals run review-first. Flipping one to full-auto is a one-line database change; ask for it when a vertical has earned trust.

Approval semantics: approving a BRIEFING publishes it. Approving a NEW_TOPIC automatically starts the Curriculum Planner on it. Rejections teach the scout what you consider noise (via your feedback loop).

6 · Subscribers & entitlements (manual billing bridge)

Stripe is not wired yet — you are the billing system until launch:

  1. Customers sign up free at /en/subscribe (works because subscribers_enabled is ON). They automatically get PREVIEW on every active vertical: truncated briefings, course plans visible, no downloads.
  2. When someone pays you (bank transfer, invoice — whatever you agree), open Subscribers and click the vertical chip on their row to flip it to FULL: complete briefings, full educator guides, all pack downloads, full digest emails.
  3. Click again to send them back to PREVIEW (non-payment, refund).
At launchStripe replaces your clicks: per-vertical monthly plans + the premium Investment plan drive the same Entitlement records automatically. Nothing else changes — the portal, gating, and digests already enforce entitlements.
Digest emailsResend is in test mode: it only delivers to your own address (mbinmahfooz74@gmail.com) until you verify a sending domain. Digest attempts to other addresses are recorded per-subscriber in the event log with the exact reason.

7 · B2B workspaces (your GCC pilots)

A workspace is a company's private Lesson Forge: their own verticals, sources, courses, and packs — branded with their name and color — fully isolated from you and from other tenants.

Onboarding a pilot company

  1. Send them /en/workspace/new (works because b2b_enabled is ON). They enter company name, admin name/email, password.
  2. Their admin invites colleagues from their Workspace page — invite emails carry an accept link; seat limits are enforced (default 5).
  3. They create their own vertical (self-builds like yours), add sources, plan courses, and generate packs carrying their branding.
  4. You watch it all from Tenants: users, verticals, seats per workspace.
Current limitationsTenant brand color and seat limit have no UI yet — they're set in the database (the Falcon demo tenant uses teal #0E7490, 3 seats). Ask and I'll change them, or a settings UI can be added. Workspace billing is manual/invoice for pilots.

As platform owner you can see every tenant's data (support access). Tenants can never see yours or each other's — enforced on every page, action, and file download.

8 · The engine room (runbook)

Start / restart

# 1. database (skip if `docker ps` shows lessonforge-db)
npm run db:up

# 2. engine worker — Git Bash (sources your API keys, then starts)
set -a && source .env && set +a && npm run dev:worker

# 3. web app — second terminal
npm run dev

Worker is healthy when it prints: Queues: ping, agent.run, source.ingest, weekly.cycle (cron Mon 06:00 UTC). Restart the worker (Ctrl+C, rerun) after any .env change — it hot-reloads code but not environment variables.

Health checks

# queue round-trip (worker must be running)
npm run ping --workspace=packages/engine

# run any agent by hand
npm run agent --workspace=packages/engine -- curriculum-planner ai-emerging-tech '{"title":"Prompt engineering","sessions":4}'

Quality gate (run before shipping any engine change)

npm run eval --workspace=packages/engine

Five golden topics are planned + drafted and scored on a structural rubric. Fails below 0.60 or on a >0.10 drop vs the last run from the same provider (Groq baseline: 0.899). CI runs this on every push too.

Backups

npm run db:backup   # timestamped pg_dump into backups/, keeps latest 14
Do this weeklyBackups are manual right now. One Docker-volume loss erases everything. Friday backup, always.

10 · Troubleshooting

SymptomFix
Docker Desktop won't start (known issue on this machine)PowerShell: kill Docker processes → Rename-Item "$env:LOCALAPPDATA\Docker\run" "run_stale_$(Get-Random)" → relaunch Docker Desktop, wait 2–3 min. Root cause (Docker AI socket) is already disabled in settings.
Agent output says [fallback]The worker has no LLM key in its environment. Restart it with the .env sourcing line from §8.
Sources stuck in QUEUEDWorker isn't running or crashed — restart it; jobs resume automatically from the queue.
Source FAILEDOpen it — the error lists every extraction method tried. Paywalled/blocked pages sometimes work via a different URL of the same content. Retry button re-queues.
Digest "not sent"Resend test-mode restriction (§6) or missing key — the per-subscriber event records the exact reason.
Groq rate-limit warnings in worker logNormal on free tier — the engine waits 20s and retries once, then falls back gracefully. Heavy testing days may hit it more often.
Login works but session drops immediatelyOnly affects http through certain proxies — use http://localhost:3000 directly in your own browser.
